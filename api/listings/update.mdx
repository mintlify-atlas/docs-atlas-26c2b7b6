## Overview

Updates an existing listing. Requires authentication and ownership verification.

<Warning>
  After updating, the listing's `isApproved` status is automatically set to `false` and requires re-approval by an admin.
</Warning>

## Authentication

**Required:** Yes (JWT token in cookies)

## Authorization

Only the listing creator or an admin can update a listing. This is enforced by the `checkOwnership` middleware.

## Path Parameters

<ParamField path="id" type="string" required>
  The MongoDB ObjectId of the listing to update
</ParamField>

## Content Type

```
multipart/form-data
```

(Optional, only if uploading new images)

## Examples

### Update listing details

```bash
curl -X PATCH http://localhost:5000/api/listings/507f1f77bcf86cd799439011 \
  -H "Cookie: jwt=YOUR_JWT_TOKEN" \
  -F "title=Updated Luxury Villa" \
  -F "description=Newly renovated with modern amenities" \
  -F "price=2800000" \
  -F "location[city]=dubai" \
  -F "location[area]=Dubai Marina" \
  -F "propertyType=villa" \
  -F "bedrooms=4" \
  -F "bathrooms=4" \
  -F "areaSqft=3800"
```

### Update with additional images

```bash
curl -X PATCH http://localhost:5000/api/listings/507f1f77bcf86cd799439011 \
```

## Implementation Details

- Implemented in `controllers/listingController.js:191`
- Route defined in `routes/listingRoutes.js:29`
- Uses `checkOwnership` middleware to verify user is the creator or an admin
- New images are **appended** to the existing images array (not replaced)
- The `status` field cannot be changed through this endpoint
- After any update, `isApproved` is set to `false` requiring admin re-approval
- Missing fields in the request retain their previous values as fallback
